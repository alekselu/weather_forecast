# Анализ доступных данных Open-Meteo и оценка объемов хранения

## 1. Обзор доступных данных Open-Meteo

API Open-Meteo предоставляет два ключевых уровня агрегации данных:

### 1.1. Daily (дневные агрегаты)

Агрегируются из почасовых данных и включают:

- Температура:
  - temperature_2m_mean / min / max
  - apparent_temperature_*
- Осадки:
  - precipitation_sum, rain_sum, snowfall_sum
  - precipitation_hours
- Атмосфера:
  - pressure_msl_mean
  - relative_humidity_2m_mean
- Ветер:
  - wind_speed_10m_max
  - wind_gusts_10m_max
- Радиация:
  - shortwave_radiation_sum
  - sunshine_duration
- Дополнительно:
  - uv_index_max
  - daylight_duration
  - weather_code

### 1.2. Hourly (почасовые данные)

- temperature_2m
- relative_humidity_2m
- dew_point_2m
- apparent_temperature
- pressure_msl
- precipitation
- wind_speed / direction
- radiation, visibility и др.

## 4. Оценка объема хранения (PostgreSQL)

Предположения:

- 1 строка:
  - timestamp (8 байт)
  - city_id (4 байта)
  - ~10 float колонок (~80 байт)
  - служебные данные (~30–50 байт)

Итого:
≈ 120–150 байт / строка

---

### 4.1. Daily

| Масштаб | Строк | Объем |
|--------|------|-------|
| 1 город, 3 года | ~1k | ~150 KB |
| 10 городов | ~11k | ~1.5 MB |
| 50 городов | ~55k | ~7–8 MB |

---

### 4.2. Hourly

| Масштаб | Строк | Объем |
|--------|------|-------|
| 1 город, 3 года | ~26k | ~3–4 MB |
| 10 городов | ~260k | ~30–40 MB |
| 50 городов | ~1.3M | ~150–200 MB |

---

### Вывод по хранению:

- Daily — практически бесплатно
- Hourly — умеренно дорого, но в пределах нормы
- До 10 млн строк — безопасный предел (≈1–1.5 GB)

## 6. Сравнение daily vs hourly

| Критерий | Daily | Hourly |
|----------|------|--------|
| Объем | низкий | высокий |
| Шум | ниже | выше |
| Простота модели | высокая | ниже |
| Feature engineering | ограничен | богатый |
| Точность | средняя | выше потенциал |

## 7. Итоговые схемы

## 7.1. Реалистичная схема (рекомендуется)

- Города: ~10
- Период: 3–5 лет
- Данные: daily
- Признаки: ~10–12

Итого:

- ~15 000 строк
- < 5 MB

Плюсы:

- простота
- быстрое обучение
- минимальная нагрузка

---

## 7.2. Максимальная схема

- Города: до 50
- Период: 10 лет
- Данные: hourly
- Признаки: ~15–20

Итого:

- ~4–5 млн строк
- ~500–700 MB

Плюсы:

- максимальная точность
- богатый feature space

Минусы:

- сложность
- нагрузка на обучение
- необходимость оптимизации (индексы, партиции)

---

## 8. Итоговые выводы

1. Open-Meteo предоставляет достаточный набор признаков для построения модели без внешних источников.
2. Для v1 системы:
   - daily данные полностью достаточны
3. Hourly данные:
   - дают прирост точности
   - увеличивают сложность и объем в 20–30 раз
4. PostgreSQL:
   - выдерживает все рассматриваемые сценарии без шардинга
5. Оптимальная стратегия:
   - старт с daily
   - переход к hourly при необходимости

---

## 9. Стратегия

Начать с:

- daily данных
- ~10 признаков
- 3–5 лет истории
- ~10 городов

Далее:

- добавить hourly данные при необходимости предсказывать данные на конкретный час, а не в среднем на день
- перейти к мультитаргетной модели
