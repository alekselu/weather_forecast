# Выбор ML модели для прогнозирования погоды (расширенный анализ)

## 1. Анализ данных (Open-Meteo)

### 1.1. Объем и структура данных

API Open-Meteo предоставляет:
- данные с 1940 года
- временное разрешение:
  - hourly (почасовые)
  - daily (дневные)
- десятки целевых параметров (температура, осадки, ветер, давление, различные статистики параметров и др.)

Общий объем исходных данных:
- полный датасет ≈ 90 ТБ (глобально)

Для системы используется **подвыборка**:

### 1.2. Реальный объем данных в системе

Для 1 города:

| Тип данных | Размер |
|------------|--------|
| Daily (3 года) | ~1095 строк |
| Daily (10 лет) | ~3650 строк |
| Hourly (3 года) | ~26 000 строк |
| Hourly (10 лет) | ~87 000 строк |

При масштабировании:
- 50 городов × 10 лет (daily) → ~180k строк
- 50 городов × 10 лет (hourly) → ~4 млн строк

Вывод:
- данные **не являются большими на уровне системы** - поступающий объем новых данных мал по сравнению с хранимыми записями
- узкое место — не объем, а **шум и сложность зависимости**

---

### 1.3. Характер данных

- выраженная сезонность - смена времен года и соответствующих режимов => отсутствие стационарности ряда
- слабый тренд
- высокая стохастичность
- нелинейные зависимости
- влияние внешних факторов (которые не всегда наблюдаемы)

Ключевой вывод:
> погодные данные — это хаотический временной ряд

---

## 2. Обзор  подходов

## 2.1. SARIMA / SARIMAX

### Особенности

SARIMA:

- учитывает сезонность через (P, D, Q, s)
- позволяет работать с нестационарными рядами

SARIMAX:

- расширение SARIMA с экзогенными признаками

Формально:

```

SARIMA(p, d, q)(P, D, Q, s)

```

### Плюсы

- встроенная работа с временной структурой
- математическая интерпретируемость
- поддержка сезонности

### Минусы

- требуется подбор параметров (grid search)
- плохо масштабируется на множество рядов - городов и предсказываемых параметров
- чувствительна к шуму - требуется регуляризация
- сложность добавления большого числа признаков
- отдельная модель на каждый город - увеличение стоимости дообучения

---

## 2.2. XGBoost (градиентный бустинг)

Подход:

```

y(t) = f(lag_features, calendar_features, exogenous_features)

```

### Плюсы

- не требует стационарности
- устойчив к шуму
- легко масштабируется
- поддерживает мультитаргетные предсказания
- легко добавлять признаки
- быстро обучается

### Минусы

- нет встроенного понимания времени
- требует feature engineering для экстраполяции трендов

---

## 2.3. Darts (фреймворк временных рядов)

Darts —  библиотека, объединяющая:

- классические модели (ARIMA, SARIMA)
- ML (XGBoost, LightGBM)
- DL (RNN, Transformer)

### Особенности

- единый API для временных рядов
- поддержка мультитаргетных временных рядов
- встроенные пайплайны

### Плюсы

- ускоряет разработку
- поддерживает разные типы моделей
- удобен для экспериментов

### Минусы

- является не моделью, а оберткой
- увеличивает сложность системы
- ограниченный контроль на низком уровне

---

## 3. Особенности специализированных моделей (SARIMA) в рамках задачи

### 3.1. Учет нестационарности

- SARIMA учитывает сезонность
- устраняет нестационарность через дифференцирование
- погодные данные содержат:
  - резкие скачки
  - нелинейные эффекты
  - шум

SARIMA предполагает:

- линейность
- стабильность структуры

Вывод:
> формально SARIMA корректна, но практически — часто нестабильна

---

### 3.2. Масштабируемость (критичный фактор)

| Критерий | SARIMA | XGBoost |
|----------|--------|--------|
| Модель на город | 1 модель / город | 1 модель на все города (возможно) |
| Автонастройка | сложная | простая |
| Batch обучение | ограничено | легко |
| Расширение | сложно (по модели на каждый временной ряд) | просто |

Вывод:
> SARIMA плохо масштабируется при росте числа городов

---

### 3.3. Дообучение

Требование: еженедельное обновление

SARIMA:

- нет полноценного incremental learning
- часто требуется переобучение + подбор параметров

XGBoost:

- быстрое переобучение с нуля
- стабильное время обучения

Вывод:
> XGBoost лучше соответствует требованиям CI/CD

---

### 3.4. Работа с признаками

SARIMA:

- базово — только временной ряд
- SARIMAX добавляет признаки, но:
  - усложняет модель
  - ухудшает устойчивость

XGBoost естественно работает с:

- лагами - **при добавлении соответствующих признаков в процессе feature engineering**
- календарными признаками
- дополнительными параметрами

---

## 4. Итоговое сравнение

| Критерий | XGBoost | SARIMA / SARIMAX | Darts |
|----------|--------|------------------|-------|
| Тип модели | ML (регрессия) | статистическая | фреймворк |
| Учет времени | через признаки | встроенный | зависит от модели |
| Нестационарность | не требует | учитывается | зависит |
| Устойчивость к шуму | высокая | низкая | зависит |
| Масштабируемость | высокая | низкая | средняя |
| Мультитаргетность | да | нет | да |
| Добавление признаков | просто | сложно | просто |
| Дообучение | простое | сложное | зависит |
| Скорость обучения | высокая | средняя | зависит |
| Production readiness | высокая | средняя | ниже |
| Гибкость | высокая | низкая | высокая |

---

## 5. Вывод

Несмотря на то, что SARIMA и SARIMAX формально лучше подходят для временных рядов и учитывают сезонность и нестационарность, они **не являются оптимальным выбором в данной задаче**.

Причины:

1. Погодные данные:
   - шумные
   - нелинейные
   - плохо соответствуют предположениям статистических моделей

2. Требования системы:
   - масштабируемость (много городов)
   - расширяемость (новые параметры)
   - регулярное переобучение

3. Инженерные ограничения:
   - необходимость простой интеграции
   - ограниченные ресурсы

Итог:

> XGBoost является более практичным и устойчивым решением для production-системы прогнозирования погоды.

---

## 6. Рекомендация

Использовать:

- XGBoost / LightGBM
- lag features + календарные признаки
- возможность перехода к мультитаргетной модели

Darts:

- использовать  как инструмент для экспериментов и запуска / сравнения моделей

SARIMA / SARIMAX:

- можно использовать как baseline или для сравнения
